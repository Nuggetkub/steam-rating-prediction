========================================================================
  THE SINGLE-GAME DEVELOPER PROBLEM
  Pre-Split Mean Target Encoding — Issue, Impact & All Failed Solutions
========================================================================
  Project : Steam Rating Prediction
  Dataset : 22,065 games (after MIN_REVIEWS=100 filter)
  Date    : 2026-05-08
  Model   : XGBoost (Optuna TPE, 50 trials x 5-fold CV)
========================================================================


── THE CORE STRUCTURAL PROBLEM ─────────────────────────────────────────

Out of 22,065 games in the training pool:
  - Total unique developers    : ~19,600+
  - Developers with 1 game     : ~9,639  (44% of all developers)
  - Developers with 2 games    : ~3,000+ (est.)
  - Developers with <3 games   : ~55%+ of all developers
  - Developers with <5 games   : ~60%+ of all developers

developer_rating_mean is the #1 most important feature in the model.
  - Feature importance (XGBoost gain): 22–34% across all runs
  - Without it or with it degraded, all models collapse toward global_mean


── WHAT MEAN TARGET ENCODING DOES ──────────────────────────────────────

Standard pre-split mean target encoding:

  developer_rating_mean[game X] = mean(Rating of ALL games by same developer)

For a developer with MULTIPLE games:
  e.g. Valve has Portal, Portal 2, Half-Life 2, CS:GO ...
  developer_rating_mean ≈ stable average across many titles (~8.5+)
  Any single game's contribution is small → low leakage risk

For a developer with ONE game (44% of all devs):
  developer_rating_mean[game X] = Rating[game X]  (exactly)
  The feature equals the target — this is direct target leakage.

  Example:
    "Obscure Dev" made exactly 1 game with Rating = 6.23
    developer_rating_mean = 6.23  ← same as target
    XGBoost sees this as a perfect predictor for that row


── WHY THIS IS "LOAD-BEARING LEAKAGE" ───────────────────────────────────

In most ML problems, target leakage is purely harmful and should be
fixed. This dataset is an exception because of the following:

  1. DEPLOYMENT CORRECTNESS:
     At real prediction time, we are predicting a game that does not
     exist yet. The developer's past games ALL exist and are known.
     Using the full developer mean is not leakage in deployment —
     it is using legitimately available historical information.

  2. THE SIGNAL IS REAL:
     A developer who made a 9.5-rated game is genuinely more likely
     to make another high-quality game. Even for single-game devs,
     their one past game IS the best available predictor.
     Discarding it does not reduce leakage — it destroys signal.

  3. EVALUATION METRIC IMPACT:
     The TEST RMSE (0.5605) is slightly optimistic because test games
     contribute to their own developer's mean.
     The HOLD-OUT RMSE (1.1133) is the clean metric — 32 games were
     removed from df_clean BEFORE any encoding was computed.
     The hold-out is the honest evaluation number to report.


── ALL SOLUTIONS ATTEMPTED AND WHY THEY FAILED ─────────────────────────

────────────────────────────────────────────────────────────────────────
FIX 1: Post-Split Target Encoding
────────────────────────────────────────────────────────────────────────
  Idea:
    Compute developer_rating_mean using ONLY df_train (after split).
    Test/hold-out games use train-only means. Unknown devs → global_mean.

  What happens to single-game devs:
    If a dev's only game lands in the TEST set (20% chance per game):
      → Their game is not in df_train
      → developer_rating_mean = global_mean (7.97) for that test game
    If their only game is in TRAIN:
      → Still leaky (mean = 1 game = its own rating), just less often

  Result: CATASTROPHIC
    Test RMSE:  0.6336 → 1.29   (+0.66, more than doubled error)
    Root cause: ~44% of developers have their ONLY game randomly
                assigned to the test set 20% of the time, all
                collapsing to global_mean. The model trained on
                informative individual means but evaluated on mostly
                global_mean — massive distributional shift.

────────────────────────────────────────────────────────────────────────
FIX 2: K-Fold OOF (Out-Of-Fold) Target Encoding
────────────────────────────────────────────────────────────────────────
  Idea:
    Use category_encoders.TargetEncoder with cv=5.
    Each game's developer mean is computed from the other 4 folds.
    Prevents the model from seeing its own target during training.

  What happens to single-game devs:
    A dev with 1 game appears in exactly 1 fold.
    In 4 out of 5 folds (where they are held out), their mean is
    computed from 0 samples → falls back to global_mean (7.97).
    During training, this game always trains on global_mean.

  Result: CATASTROPHIC
    Test RMSE:  0.6339 → 1.1208  (+0.49)
    R²:         0.81   → 0.41    (model barely better than mean)
    Root cause: 9,639 developers (44%) have exactly 1 game.
                OOF collapses ALL of them to global_mean.
                developer_rating_mean loses all individual signal
                for nearly half the dataset.

────────────────────────────────────────────────────────────────────────
FIX 3: Bayesian Smoothing (pre-split, m=1)
────────────────────────────────────────────────────────────────────────
  Idea:
    smoothed_mean = (n × raw_mean + m × global_mean) / (n + m)
    where m=1, n = number of games by that developer.
    Small n → pulls toward global_mean. Large n → stays near raw_mean.

  What happens:
    Single-game dev with rating 9.5:
      smoothed = (1×9.5 + 1×7.97) / (1+1) = 8.74  (pulled down)
    Single-game dev with rating 4.0:
      smoothed = (1×4.0 + 1×7.97) / (1+1) = 5.99  (pulled up)
    Developer range compressed from [1.06–10.00] → narrower band

  Result: WORSE
    The developer signal range compressed significantly.
    The smoothed means diluted the real variation between good and
    bad developers, making it harder for XGBoost to distinguish them.

────────────────────────────────────────────────────────────────────────
FIX 4: Bayesian Smoothing (pre-split, m=5)
────────────────────────────────────────────────────────────────────────
  Idea: Same as Fix 3 but stronger smoothing weight m=5.

  Result: WORSE than m=1
    Higher weight = more compression = even less signal.
    All 5 pre-split smoothing variants (m=1, m=3, m=5, m=7, m=10)
    were tested during early experiments. All were worse than raw means.

────────────────────────────────────────────────────────────────────────
FIX 5: Bayesian Smoothing (post-split, weight=15)
────────────────────────────────────────────────────────────────────────
  Idea:
    Apply strong Bayesian smoothing after train/test split.
    weight=15 means a dev needs 15 games to be trusted at full mean.

  What happened:
    Developer mean range: [1.06–10.00] → [6.47–8.75]
    Nearly all developers collapsed to within 1.5 points of global_mean.
    The model could no longer distinguish a great dev from a bad one.

  Result: CATASTROPHIC
    Baseline RMSE: 0.63 → 1.20  (nearly doubled)
    The compression was so severe the feature became useless.

────────────────────────────────────────────────────────────────────────
FIX 6: Bayesian Smoothing + First-Time Developer Grouping
         (post-split, weight=3)
────────────────────────────────────────────────────────────────────────
  Idea:
    Two-part fix:
    (a) Group devs with <2 games in training as "First_Time_Dev" (FTD)
        → all get a single shared FTD mean
    (b) Apply Bayesian smoothing (weight=3) to remaining devs

  What happened:
    8,472 developers (48%) had <2 training games → all became FTD
    FTD group mean ≈ global_mean (because average across all newcomers)
    developer_rating_mean feature importance: 38% → 5%
    The model effectively lost the developer signal for half the data.

  Result: CATASTROPHIC
    Test RMSE:   0.6108 → 0.6851  (+0.074)
    Hold-out R²: +0.22  → −0.03   (model WORSE than predicting the mean)
    CV/test gap: widened by +0.063 (severe overfitting signal)
    5 total smoothing variants tested. All conclusively worse.
    CLOSED — not applicable for this dataset structure.

────────────────────────────────────────────────────────────────────────
FIX 7 (PROPOSED, NOT TESTED): Rare Category Grouping
────────────────────────────────────────────────────────────────────────
  Idea (suggested by collaborator):
    Bundle all developers with <3 games (or <5 games) into a single
    "Rare_Developer" category before mean encoding.
    "Rare_Developer" gets one shared mean. Others keep individual means.

  Why this will fail (equivalent to Fix 6):
    With threshold=3: ~55%+ of all developers → "Rare_Developer"
    "Rare_Developer" mean = average over thousands of diverse devs
                          ≈ global_mean (7.97)
    Result: 55% of developers lose all individual signal.
    A dev with one 9.5 game and a dev with one 4.0 game both map to 7.97.
    XGBoost cannot distinguish them.

    This is structurally identical to Fix 6 (FTD grouping with <2),
    but MORE aggressive (threshold 3 vs 2).
    Fix 6 failed catastrophically. Method 1 would be strictly worse.

  With threshold=5: ~60%+ collapse. Even worse.

  NOT TESTED — prior experiments conclusively cover this case.

────────────────────────────────────────────────────────────────────────
FIX 8 (PROPOSED, NOT TESTED): Leave-One-Out Encoding (LOOE)
────────────────────────────────────────────────────────────────────────
  Idea (suggested by collaborator):
    For each training game, compute developer mean EXCLUDING that game.
    For single-game devs: mean from 0 games → global_mean.
    For test/holdout: use train-only means (no LOOE at inference).

  Why this will fail (equivalent to Fix 2):
    Single-game developer (44% of all devs):
      Training: 1 game excluded → 0 remaining → mean = global_mean
      This is IDENTICAL to K-fold OOF encoding (Fix 2) for n=1.

    Fix 2 (K-fold OOF) crashed test RMSE from 0.63 → 1.12, R² 0.81→0.41.
    LOOE produces the same result for single-game devs.
    Multi-game devs are slightly different (LOOE vs K-fold), but the
    44% single-game dev collapse dominates and causes the same crash.

  NOT TESTED — Fix 2 conclusively covers this case.


── ROOT CAUSE SUMMARY ───────────────────────────────────────────────────

Every attempted fix fails for the same reason:

  Any method that withholds or groups a single-game developer's own
  rating during training replaces an informative individual signal
  (that dev's actual quality level) with global_mean (7.97).

  This affects 44% of all developers simultaneously.
  XGBoost detects this as: "44% of developers all have quality = 7.97"
  and cannot learn to distinguish them during training.

  At evaluation time, those same developers either:
    (a) have their real mean applied → distributional shift → crash
    (b) also get global_mean → model predicts near-7.97 for 44% of games

  Either way, the model performs worse than the "leaky" version.


── CURRENT ACCEPTED APPROACH ────────────────────────────────────────────

  Pre-split mean target encoding (no correction) is kept because:
  1. All alternatives tested were significantly worse
  2. At deployment, full developer history is known — no real leakage
  3. The #1 feature (developer_rating_mean) retains full signal
  4. The honest evaluation metric is the HOLD-OUT RMSE, not test RMSE

  The test RMSE (0.5605) is acknowledged as slightly optimistic.
  The hold-out RMSE (1.1133, 32 games, removed before any encoding)
  is the correct number to report in academic or production contexts.

  Current model metrics (2026-05-08):
    Test-set : RMSE 0.5605 | MAE 0.3076 | R² 0.8539  (slightly optimistic)
    Hold-out : RMSE 1.1133 | MAE 0.8253 | R² 0.1223  (honest metric)
    Hold-out : Accuracy 56% | within 0.5 pts 16/32 (50%)


── WHAT WOULD ACTUALLY HELP ─────────────────────────────────────────────

The real problem is not the encoding method — it is that single-game
developers have only 1 data point, making any representation of their
"quality" inherently uncertain. Better data would help more than better
encoding:

  1. External data sources:
     - Metacritic/OpenCritic scores for more games
     - Itch.io developer history for indie studios
     - Game jam participation as a proxy for smaller devs

  2. Accept the uncertainty explicitly:
     - Confidence intervals / quantile regression
     - Flag is_single_game_dev as a binary feature (already have
       developer_game_count, which captures this)

  3. Report the hold-out honestly:
     - Test RMSE is a leaky metric for this dataset
     - Hold-out RMSE is the correct benchmark to cite
